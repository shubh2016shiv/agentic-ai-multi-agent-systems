# Chapter 2 — Core Concepts: The Data Contracts

> **Learning chapter** — An explanation of why the `models.py` file is the most important file in the Orchestration module.

---

## 2.1 The Concept of a Data Contract

In a monolithic Python script, if `Function A` returns a dictionary and `Function B` expects a list, the code crashes instantly on your laptop. You fix it, and you move on.

In a **Multi-Agent System**, data is passed between independent, non-deterministic LLMs. If the Pulmonologist agent decides to return a markdown string, and the Cardiologist agent decides to return JSON, the Synthesis agent will fail trying to parse them.

To prevent this, we enforce **Data Contracts**: rigidly defined structures that *guarantee* what data looks like, regardless of which agent produced it or which orchestration architecture is running.

---

## 2.2 `OrchestrationResult` (The Envelope)

Every single specialist agent in every one of the 5 orchestration architectures MUST return an `OrchestrationResult`. 

```python
class OrchestrationResult(BaseModel):
    agent_name: str
    specialty: str
    output: str 
    duration_seconds: float
    was_successful: bool
    error_message: Optional[str]
```

### Why this design?
1. **The Envelope Pattern:** The `output` string is the actual clinical advice generated by the LLM. The rest of the fields are the "envelope" used by the router/synthesizer.
2. **Identification:** The `agent_name` and `specialty` fields tell the Synthesizer *who* said *what*. Without this, the Synthesizer wouldn't know if the lung advice came from a junior nurse or the Chief of Pulmonology.
3. **Resilience Tracing:** The `was_successful` and `error_message` fields allow the graph to gracefully handle failures. If the Cardiology API crashes, `was_successful = False`. The Synthesizer can read this envelope and say, "I am writing the final report, but please note Cardiology was unavailable."

## 2.3 `PatientWorkload` (The Payload)

The `PatientWorkload` is what the user (or the API endpoint) submitting the task pushes into the start of the graph.

```python
class PatientWorkload(BaseModel):
    patient_case: PatientCase
    required_specialties: list[str] 
    urgency_level: str 
    orchestration_notes: str 
```

### Why not just pass the `PatientCase`?
A `PatientCase` contains raw medical data (age, symptoms, vitals). It knows nothing about the Multi-Agent System.

The `PatientWorkload` wraps the raw medical data with **Routing Metadata**:
1. `required_specialties`: Tells the *Dynamic Router* which agents to wake up.
2. `urgency_level`: Tells the *Supervisor* whether to use the expensive, fast LLM or the cheaper, slow LLM.
3. `orchestration_notes`: Free-text instructions from the human operator ("*Please focus heavily on the kidney interaction*").

## 2.4 The `SHARED_PATIENT` Constant

If you open `models.py`, you will see a hardcoded `SHARED_PATIENT` (a 68-year-old male with COPD and Heart Failure).

*Why hardcode test data inside a core module?*

This repository is designed for **education**. If the Supervisor uses Patient A, and the Dynamic Router uses Patient B, it is impossible for a developer to compare the outputs and see which architecture performed better.

By enforcing that all 5 pattern scripts (`scripts/orchestration/`) import and process the exact same `SHARED_PATIENT`, the developer can run all 5 scripts and visibly see how the *Orchestration Architecture* changes the final output, isolating the variable of the patient data perfectly.
